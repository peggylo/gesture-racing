# 學習筆記 — 手勢操控賽車專案

記錄這次專案中學到的概念。

## 通訊協定

### UDP vs TCP vs File

三種程式間傳遞資料的方式，差異在於可靠性與複雜度：

- **UDP**（對講機）：送出去就不管了，不確認對方有沒有收到。快、簡單，適合即時性高但丟幾筆沒差的場景（如手勢資料一秒送 30 次）
- **TCP**（打電話）：先三次握手建立連線，每筆資料都確認收到，沒收到就重送。可靠但多了連線管理的複雜度
- **File/Pipe**（留紙條）：寫檔案讓對方讀，最直觀但延遲高

### TCP 三次握手

建立連線的過程：A 說「我要跟你講話」→ B 回「好，準備好了」→ A 再回「OK，開始」。傳輸中每筆資料都有編號和收到確認（ACK），斷線時也有對應的結束流程。

## 鍵盤模擬

### pynput 的運作原理

pynput 透過 macOS 系統層級 API 注入鍵盤事件，產生的事件跟實際按鍵盤完全一樣，應用程式無法區分。因此需要「輔助使用」權限，也因此按鍵會送到當前焦點視窗而非指定應用程式。

### 差異更新（Differential Update）

用 set 追蹤目前被按住的按鍵，每次新手勢進來時比對「該放開哪些、該按下哪些」，避免重複按下或遺漏放開導致按鍵卡住。

## 手勢辨識

### MediaPipe Hand Landmarks

MediaPipe 從 2D 影像推估出手部 21 個關節點的座標，每個點有 x、y、z 三個值。但 z 軸只是以手腕為基準的相對深度，精度遠不如 x、y，因此稱為 **2.5D**（不是完整的 3D 空間重建）。

每根手指有 4 個點：

- **MCP**：手指根部（與手掌連接的關節）
- **PIP**：第二個關節
- **DIP**：第三個關節
- **TIP**：指尖

### 手指伸直判斷

指尖的 y 座標 < PIP 關節的 y 座標（螢幕座標 y 向下為正，指尖在上方 = y 值較小）= 手指伸直。

### 手勢辨識的限制

MediaPipe 在手部側面、手指重疊時辨識不穩定（如刀手姿勢），因為深度資訊不足，關節點會擠在一起。設計手勢時應選擇關節點分布差異大的姿勢（如握拳 vs 張開手掌）。

## 架構設計

### 模擬按鍵 vs 直接整合

本專案用「模擬鍵盤按鍵」的方式操控遊戲，好處是遊戲完全不用改，壞處是只能在本機跑。如果要讓其他人遠端玩，應該把手勢辨識搬到瀏覽器端（MediaPipe JS SDK），直接在遊戲程式碼裡接收手勢事件。

### 通訊介面先行

拆成兩支程式時，先定義好通訊格式（UDP port + JSON schema），兩邊就能獨立開發不會打架。介面規格是唯一的交會點。
